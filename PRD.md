## 1. 專案概述 (Overview)

### 1.1 核心設計理念

本專案為支援 **多店點/連鎖架構** 的綜合場館管理系統。核心設計理念為：

| 理念 | 說明 |
|------|------|
| **會員帳號通用化 (Universal ID)** | 會員帳號在底層是唯一的，支援未來擴充接入其他預約系統或不同服務體系（如：皮拉提斯、物理治療），資料可互通 |
| **總部/分店分層管理** | 總部可查看所有分店數據、跨店報表、統一制定規章；分店僅能查看本分店數據 |

系統需整合 **CRM 客戶管理**、**電子合約**、**HR 薪資排班**、以及 **多維度的財務營運報表**。

### 1.2 背景

目前的健身房管理軟體多為舊式單體架構，難以擴充且缺乏現代化的使用者體驗。常見痛點包括：

- **資料孤島**：各分店數據無法統一彙整，總部難以掌握全局
- **人工作業繁瑣**：合約請假/轉讓需人工計算到期日，易出錯
- **權限管理粗糙**：無法細緻控制「教練只能看自己學員」等場景
- **報表滯後**：財務數據需事後人工統計，無法即時決策
- **會員體驗差**：缺乏自助服務 App，預約/查詢需臨櫃處理
- **教練管理分散**：執課紀錄、學員追蹤無統一系統

### 1.3 解決方案

構建一套支援 **多場館 (Multi-branch)**、**高度自動化** 且 **數據驅動** 的 SaaS 系統：


### 1.4 系統架構邏輯

#### 多店架構 (Multi-Store Support)

| 類型 | 說明 | 權限範圍 |

### 1.5 目標

| 類型 | 目標 | 衡量指標 |
|------|------|----------|
| **擴展性** | 支援 50+ 間分店數據量 | 報表查詢 < 2 秒 |
| **自動化** | 合約異動自動計算到期日 | 人工錯誤率 < 1% |
| **精準權限** | 多層權限分級 (7 種角色) | 資料外洩事件 = 0 |
| **即時報表** | 財務/HR/行銷自動統計 | 報表產出時間 < 1 分鐘 |
| **會員體驗** | App 自助服務 | 臨櫃作業減少 50% |
| **教練效率** | 統一執課管理 | 紀錄完整率 > 95% |

### 1.6 範圍界定

#### In Scope (MVP + Phase 2)
- 多分店組織架構與權限
- 會員 CRUD 與標籤系統
- 合約生命週期管理 (含請假/轉讓)
- 財務收款紀錄與報表
- HR 打卡、排班、休假、薪資
- **會員端 App** (預約、查詢、目標追蹤)
- **教練端 App** (學員管理、執課紀錄)
- **行銷後台** (Leads、活動、RFM)
- **人資後台** (考核、升遷、薪資管理)

#### Out of Scope (Future)
- Wger 運動數據深度整合
- 進階 BI 分析 (AI 預測)
- Mobile Native App (暫用 PWA)
- 第三方金流串接

## 3. 功能需求 (Functional Requirements)

### 3.1 組織與權限模組 (Organization & Access)

#### FR-ORG-001: 多分店管理
| 項目 | 說明 |
|------|------|
| **描述** | 系統需支援新增分店，並設定分店屬性 |
| **分店類型** | `HEADQUARTER` (總店) / `BRANCH` (分店) |
| **分店屬性** | 名稱、地址、統編、營業時間、特定設定 (JSON) |
| **限制** | 總店僅能有一間，分店數量無上限 |

#### FR-ORG-002: 職位權限配置
| 項目 | 說明 |
|------|------|
| **描述** | 透過 `job_titles` 表定義職位及其預設權限 |
| **權限格式** | JSON 格式的 `permissions_config` 欄位 |
| **UI 呈現** | 前端以 Checkbox 方式呈現可勾選權限 |
| **覆寫機制** | 員工層級可透過 `custom_permissions` 覆寫職位預設 |

#### FR-ORG-003: 資料隔離 (Row-Level Security)
| 項目 | 說明 |
|------|------|
| **描述** | 分店員工預設只能存取 `branch_id` 與自己相同的資料 |
| **實作** | Directus Policy + PostgreSQL RLS |
| **規則** | 見下方權限規則表 |

**權限規則：**
| 角色 | 資料範圍 | Policy 條件 |
|------|----------|-------------|
| HQ_ADMIN | 全部 | `true` |
| STORE_MANAGER | 本店 | `branch_id = $CURRENT_USER.branch_id` |
| COACH | 負責會員 | `sales_person_id = $CURRENT_USER.id` |
| MARKETING | 全部會員 (唯讀) | `true` (read only) |
| HR | 本店員工 | `branch_id = $CURRENT_USER.branch_id` |

### 3.2 會員管理 (CRM & 合約中心)

*目標：全方位的會員資料庫，支援精細篩選與電子化行政。*

| 篩選類型 | 篩選項目 |

**管理功能：**
- **店長權限**：可直接將會員（及其合約）從 A 教練「移交」給 B 教練

### 3.3 合約與會籍系統 (Contracts & Plans)

#### FR-CON-001: 方案定義
| 項目 | 說明 |
|------|------|
| **期限制** | `TIME_BASED` - 月費/年費，有明確到期日 |
| **點數制** | `COUNT_BASED` - 課程包，有剩餘堂數 |
| **方案屬性** | 名稱、價格、期限/堂數、是否允許請假、是否允許轉讓 |

#### FR-CON-002: 電子合約生命週期

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│  草稿   │────►│  有效   │────►│  暫停   │────►│  過期   │
│ (DRAFT) │     │(ACTIVE) │     │(PAUSED) │     │(EXPIRED)│
└─────────┘     └────┬────┘     └────┬────┘     └─────────┘
                     │               │                ▲
                     │               └────────────────┤
                     │                                │
                     └──────────────────────────────►─┘
                                 解約 (TERMINATED)
```

| 狀態 | 說明 | 觸發條件 |
|------|------|----------|
| DRAFT | 業務建立，待簽名 | 新建合約 |
| ACTIVE | 簽名並付款後生效 | 完成簽名 + 付款 |
| PAUSED | 會員請假中 | 建立請假紀錄 |
| EXPIRED | 超過到期日 | `end_date < NOW()` |
| TERMINATED | 提前解約 | 手動操作 |

*目標：自動化算薪，連動執行報表與銷售報表。*

| 項目 | 說明 |
|------|------|
| **薪資結構** | 底薪、提成比、獎金 |

| **薪資匯出** | 產出薪資表 (Excel) |
| **權限限制** | HR 可查看，不可修改財務數據 |

**自動化算薪公式：**

```
應發薪資 = 底薪
         + 課時獎金（連動「執行報表」）
         + 銷售佣金（連動「銷售報表」）
         - 請假扣款
```

**職級差異化抽成：**

| 職級 | 課時獎金比例 | 銷售佣金比例 |
|------|-------------|-------------|
| 資深教練 | 較高 | 較高 |
| 一般教練 | 標準 | 標準 |
| 新手教練 | 較低 | 較低 |

> **開發建議**：第一階段先做「數據匯出給 Excel 算」，或者系統僅計算「業績獎金」與「課時獎金」，底薪與勞健保等透過外部薪資系統或 Excel 處理，最後再回填總額，以降低開發初期風險。

#### FR-FIN-002: 五大核心報表 (Core Reports)

| # | 報表名稱 | 定義 | 內容欄位 |

## 8. 架構師的技術建議 (Technical & Data Strategy)

### 8.1 資料庫設計的關鍵 - Tenant ID (Store_ID)

> **核心原則**：所有的資料表（會員、合約、員工、交易紀錄）都必須包含 `Store_ID`（或 `branch_id`）欄位。

這樣在後端撈取數據時，才能輕易做到：
- **總店看全部**：`Store_ID = All` (或不加條件)
- **分店看自己**：`Store_ID = '01'`

```sql
-- 分店店長查詢範例
SELECT * FROM members WHERE branch_id = $CURRENT_USER.branch_id;

-- 總部管理員查詢範例
SELECT * FROM members; -- 無條件限制
```

### 8.2 會員全域性 (Global User)

建議設計 `Users` 表（存帳號密碼、Universal ID）與 `Members` 表（存各店的會籍資料）。

```
┌─────────────────────────────────────────────────────────────┐
│                        資料庫設計                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────┐                                           │
│   │   Users     │  ← 全域帳號 (Universal ID)                │
│   │  - id       │                                           │
│   │  - email    │                                           │
│   │  - password │                                           │
│   └──────┬──────┘                                           │
│          │ 1:M                                              │
│          ▼                                                  │
│   ┌─────────────┐                                           │
│   │  Members    │  ← 各店會籍 (Store-specific)              │
│   │  - user_id  │                                           │
│   │  - branch_id│                                           │
│   │  - status   │                                           │
│   └─────────────┘                                           │
│                                                              │
│   → 一個 User 可對應多個 Member 紀錄                         │
│   → 例如：他在 A 店買了教練課，在 B 店買了入場券              │
│   → App 登入一次就能看到所有權益                             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 8.3 薪資計算邏輯的複雜度

> **開發風險**：薪資計算通常是最複雜的模組。

**建議的階段性實作：**

| 階段 | 範圍 | 說明 |
|------|------|------|
| **Phase 1** | 數據匯出 | 系統產出原始數據，Excel 外部計算 |
| **Phase 2** | 部分自動化 | 系統計算「業績獎金」與「課時獎金」 |
| **Phase 3** | 完整自動化 | 底薪、勞健保、加班費全自動化 |

**Phase 1 實作建議：**
- 系統僅負責計算「業績獎金」與「課時獎金」
- 底薪、勞健保等透過外部薪資系統或 Excel 處理
- 最後再回填總額至系統

這樣可以降低開發初期風險，同時保持資料完整性。

### 8.4 權限介面設計

採用 **Checkbox (勾選式)** 設計：在建立員工帳號時，除預設角色外，可手動勾選特定權限。

```
┌─────────────────────────────────────────────────────────────┐
│ 員工權限設定                                                 │

│                                                              │

│ │ [✓] 允許匯出資料                                         │ │
│ │ [ ] 允許查看全店業績                                     │ │

│ │ [ ] 允許刪除合約                                         │ │
│ │ [✓] 允許審核請假                                         │ │
│ └─────────────────────────────────────────────────────────┘ │

```
